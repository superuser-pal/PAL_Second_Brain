# Feature File Templates

This directory contains templates for feature documentation in the system-build skill.

## Format Versions

### v2 Format (Current) - Single File

**File:** `feature_template.md`

**Structure:** Single `FEATURE.md` file containing all sections:

```
FEAT_NNN_short-name/
└── FEATURE.md                    # All-in-one feature file
    ├── ## Specification          # Requirements and user stories
    ├── ## Implementation Plan    # Technical approach
    ├── ## Tasks                  # Actionable task breakdown
    ├── ## Testing Instructions   # Generated after implementation
    ├── ## Implementation Notes   # Discoveries during implementation
    └── ## Completion Summary     # Final documentation and lessons
```

**Benefits:**
- Single source of truth (one frontmatter, one status)
- Easier navigation (single file vs 4 files)
- Reduced context switching
- Token efficient (no duplicate content)
- ~500-600 lines total (vs ~900+ in v1)

**Status Flow:**
```
specified → clarified → planned → tasked → implemented → tested → documented → exported
```

**UAT Gate:** Features must pass User Acceptance Testing (UAT) before documentation. The implement workflow generates testing instructions and STOPS for user feedback. Status becomes `tested` only after user confirms tests pass.

### v1 Format (Legacy) - Four Files

**Files:**
- `spec_template.md` - Feature specification
- `plan_template.md` - Implementation plan
- `tasks_template.md` - Task breakdown
- `checklist_template.md` - Quality checklist (deprecated)

**Structure:** Four separate files in feature folder:

```
FEAT_NNN_short-name/
├── spec.md          # Requirements and user stories
├── plan.md          # Technical approach
├── tasks.md         # Actionable task breakdown
└── checklist.md     # Quality checklist (deprecated)
```

**Status:** Legacy format, maintained for backward compatibility only.

## Coexistence Strategy

Both v1 and v2 formats coexist in the system:

**Grandfather Clause:** Existing v1 features (e.g., FEAT_001) remain in 4-file format. No migration required.

**Format Detection:** All workflows automatically detect format:
- If `FEATURE.md` exists → v2 format
- If `spec.md` exists → v1 format
- Workflows handle both formats transparently

**New Features:** Always use v2 format (FEATURE.md).

## Workflow Behavior by Format

| Workflow | v1 Behavior | v2 Behavior |
| :------- | :---------- | :---------- |
| `specify` | Creates `spec.md` | Creates `FEATURE.md` with ## Specification |
| `clarify` | Updates `spec.md` | Updates ## Specification in FEATURE.md |
| `plan` | Creates `plan.md` | Appends ## Implementation Plan to FEATURE.md |
| `tasks` | Creates `tasks.md` | Appends ## Tasks to FEATURE.md |
| `implement` | Updates `tasks.md` checkboxes | Updates ## Tasks checkboxes, generates ## Testing Instructions |
| `document` | Reads spec/plan/tasks.md | Reads ## Specification/Plan/Tasks from FEATURE.md |
| `tasks_to_issues` | Reads `tasks.md` | Reads ## Tasks from FEATURE.md |

## Migration Guidance (Optional)

**Note:** Migration is NOT required. v1 features can remain in their current format.

**If you choose to migrate a v1 feature to v2:**

1. Create `FEATURE.md` from `feature_template.md`
2. Copy frontmatter from `spec.md` to FEATURE.md
3. Copy `spec.md` content to `## Specification` section
4. Copy `plan.md` content to `## Implementation Plan` section
5. Copy `tasks.md` content to `## Tasks` section
6. Leave `## Testing Instructions` empty (generated by implement workflow)
7. Leave `## Implementation Notes` and `## Completion Summary` empty
8. Archive old files to `FEAT_NNN_name/archive/` directory
9. Update frontmatter status if needed

**Warning:** Only migrate features that are not actively being worked on. Migration during implementation can cause workflow issues.

## Template Updates

When updating templates:

1. **v2 (feature_template.md):** Update the single template
2. **v1 (legacy templates):** Only update if critical bug fix needed
3. **Always:** Document changes in this README

## Questions?

- **Which format should I use?** Always use v2 (FEATURE.md) for new features.
- **What about existing features?** Leave them in v1 format. No migration needed.
- **Can workflows handle both?** Yes, all workflows detect format automatically.
- **What's the difference?** v2 consolidates 4 files into 1, reducing lines by 40% and improving navigation.

## 1. Domain Definition (MANDATORY)

A Domain is an isolated workspace directory inside the `Domains/` folder. It MUST contain a valid `INDEX.md` at its root. 

- **Directory Naming:** `PascalCase` (e.g., `BlogContent`, `PALBuilder`).
- **Path:** `Domains/[PascalCaseDir]/`

---

## 2. Core Directory Structure

Every domain SHOULD implement this 6-folder structure to maintain compatibility with standard skills:

| Folder | Purpose |
| :--- | :--- |
| `00_CONTEXT/` | General background knowledge, APIs, design specs. |
| `01_PROJECTS/` | Active structured tasks. `INDEX.md` references these. |
| `02_SESSIONS/` | Agent session logs generated via `*dismiss` command. |
| `03_PAGES/` | Reference materials and documentation. |
| `04_OUTPUT/` | Active working area, drafts, and outputs generated by agents. |
| `05_ARCHIVE/` | Deprecated, completed, or legacy data. |

---

## 3. INDEX.md: The Root Map

The `INDEX.md` file MUST exist at `Domains/[DomainName]/INDEX.md`. It acts as the routing table for domain agents.

```markdown
# [Domain Name] Index
> General description of the domain's purpose.

## Context Maps (Pointers to 00_CONTEXT)
- List critical context files and their intended purpose.

## Active Projects (Pointers to 01_PROJECTS)
- List currently active projects. Format: `[Project Name](01_PROJECTS/project_file.md) - Status`

## Data Pointers
- Key databases or asset locations.
```

---

## 4. Constraint Rules

1. **Isolation:** A domain agent CANNOT load context from a different domain unless specifically authorized by the user.
2. **Context Pointers:** When loading context, an agent MUST read `INDEX.md` FIRST, and only dive into subdirectories defined within the index. DO NOT use `cat` on entire folders.
3. **Agent Binding:** Specific domain behavior is implemented by an agent pointing its `domain:` YAML field to the domain's directory name.
